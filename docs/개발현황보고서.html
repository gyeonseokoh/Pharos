2. 프로젝트 개요 및 설계 원칙

2.1 Pharos란

Pharos는 Obsidian 플러그인 기반의 AI 프로젝트 매니저입니다. PO(프로젝트 오너)와 PM(프로젝트 멤버)이 회의·로드맵·Task·진행도를 한 곳에서 관리하고, AI 에이전트가 회의 주제 생성·자료 수집·회의록 분석 등을 보조합니다.

2.2 초기 설계 원칙 (유석 + Claude, repository-design.md 기준)

3단계 Repository 패턴 1단계: SettingsRepository (data.json)
2단계: VaultRepository (.md 파일)
3단계: HocuspocusRepository (서버 실시간 동기화)
Feature별 스키마 소유 각 feature가 자기 도메인 스키마를 직접 소유.
features/*/domain/*Schema.ts
공유 타입은 shared/types/constants.ts만.
Service = 비즈니스 로직 Facade UI · 백엔드 · AI 에이전트 모두 Service를 통해 데이터 조작.
Repository 교체 시 main.ts 한 줄만 변경.
EntityMeta 공통 규칙 모든 엔티티: version, type, id,
createdAt, updatedAt 필드 필수.

3. 설계 의도 vs 실제 구현 비교

3.1 스키마 설계 (Task 1)

설계 의도 (repository-design.md §4) 각 엔티티는 EntityMeta(version · type · id · createdAt · updatedAt)를 가져야 함. 스키마는 feature 폴더 내 domain/*Schema.ts에 위치. Zod 기반 런타임 검증 필수.

변경 전 (유석 인수 시점)

shared/types/entities.ts에 전역 타입 집중. EntityMeta 없음. enum 컨벤션 불일치: "TODO"(shared) vs "todo"(feature). availabilitySchema.ts · commitSchema.ts 미존재. Task에 linkedCommits 필드 누락.

변경 후 (현재)

모든 스키마가 각 feature의 domain/*Schema.ts에 위치. EntityMeta 적용. shared/types/entities.ts · schemas.ts 삭제 (데드코드 제거). availabilitySchema.ts · commitSchema.ts 신규 작성. TaskV1에 linkedCommits: LinkedCommitSchema[] 필드 추가. shared/types/constants.ts만 유지 (COMMIT_TASK_PATTERN · UseCase 상수).

3.2 Repository 인터페이스 (Task 2)

설계 의도 (repository-design.md §5.2) Repository 인터페이스가 "어디서 읽고 쓰는지"를 Service가 몰라도 되도록 추상화. listByTask · listByMonth · upsertBatch · watch 등 도메인 중심 메서드 정의.

주요 수정 내역

파일변경 내용
teamRepository.tsMemberRepository → TeamRepository 리네임. setStatus() 추가.
taskRepository.tslistByPhase(phaseId: string) 시그니처 수정. LinkedCommit을 schema에서 import로 변경 (로컬 정의 제거, "manual" 상태 폐기).
availabilityRepository.ts신규 작성. getByWeek · listByMember · saveSlots · deleteByWeek · watch.
commitRepository.ts신규 작성. listByTask · listByMonth · upsertBatch · deleteByMonth · watch. 초기 잘못된 시그니처(getByMonth·upsertBatch(month, commits)) 설계 의도 기준으로 수정.

3.3 Settings 기반 구현체 (Task 3)

설계 의도 (repository-design.md §6) 1단계 MVP: data.json (Obsidian settings)에 임시 저장. 나중에 VaultRepository로 교체 시 main.ts 한 줄만 변경.

신규 추가

파일내용
availabilityRepository.settings.ts주차별 가용시간. avail-{weekStart} ID 자동 생성. Listener/emit 패턴.
commitRepository.settings.ts월별 배치 저장. sha 기준 중복 제거 병합. commits-{month} ID 자동 생성.
settings.tsPharosSettings에 availabilities: Availability[] · commitBatches: CommitBatch[] 추가. PharosPluginLike에 모든 Service 등록.
main.ts모든 Repository · Service onload() 초기화 연결 완료.

3.4 Service 레이어 (Task 4)

설계 의도 UI · 백엔드 · AI 에이전트가 모두 Service만 호출. Repository를 직접 접근하지 않음. 도메인 이벤트(task:created · minutes:attached 등)를 eventBus로 발행.
Service주요 책임특이사항
TaskServiceTASK-N ID 자동 발급, userCheck 토글, appendCommiteventBus 연동 완료
RoadmapService기획/개발 로드맵 저장·조회완료
TeamService팀원 초대, 상태 변경 (active/left)완료
ProgressServiceTask 집계 (완료율 등), read-onlyv2에서 CommitRepository 연동 예정
AvailabilityService주간 가용시간 저장 (타 팀원 슬롯 보존), 공통 슬롯 조회신규 작성
CommitServicePM-4 커밋 분류 (verified/unverified), 월별/Task별 조회신규 작성. verifyResult 자동 분류 로직 포함.
ProjectService프로젝트 생성, workspaceId UUID 자동 발급crypto.randomUUID() 사용 (§1 전제 12)
MeetingsService회의록 첨부(attachMinutes), 분석 시뮬레이터 호출완료

3.5 UI → Service 연결 (Task 5, 진행 중)

설계 의도 각 ItemView는 Service만 호출. mock 데이터 직접 import 금지. settings 직접 접근 금지.

완료된 항목

항목변경 전변경 후
WeeklyAvailabilityModal weekStart = 일요일 기준. onSubmit [미구현] Notice만. weekStart = 월요일 기준 (§4.6). availabilityService.saveMemberSlots() 실 연결. plugin · memberId 주입 구조.
MinutesArchiveItemView settings.attachedMinutes 직접 저장. 레포지토리에 회의 존재 시 → meetingsService.attachMinutes(). 미존재(mock 회의) 시 → settings 폴백 (migration 과도기).
ProjectService.create() workspaceId 누락. crypto.randomUUID() 기반 ws-{UUID} 자동 발급.

미완료 항목 및 이유

RoadmapItemView mock 완전 제거 roadmapService.getPlanning()은 Roadmap 엔티티를 반환하지만, RoadmapView는 RoadmapData 뷰모델을 요구함. Roadmap → RoadmapData 어댑터 + Task 목록 로딩이 필요. 별도 작업 필요.
각 ItemView 전반 mock → Service 교체 CalendarView · MeetingsListView 등 다수 View가 meetingPageMock.ts의 하드코딩 데이터를 사용. MeetingRepository에 실제 회의 데이터가 없으므로 Service로 교체 시 빈 화면이 됨. Task 6 VaultRepository 구현 후에야 완전 해소 가능.

5. 남은 작업 및 권장사항

5.1 즉시 착수 가능 (Task 5 마무리)

RoadmapItemView 어댑터 작성 Roadmap 엔티티 → RoadmapData 뷰모델 변환 함수 작성 후 RoadmapItemView에서 roadmapService.getPlanning()·getDevelopment() 연결. settings.developmentRoadmap: RoadmapData → roadmapRepository 저장으로 이관.

5.2 Task 6: VaultRepository 구현

백엔드(Hocuspocus) 연결 직전 단계. 경석 sync 모듈과 연동 시 필요. *.vault.ts 구현체 8개 작성 + data.json → .md 마이그레이션 스크립트. 우선순위: project · meeting · task (절대 보존 등급) 먼저.

5.3 GitHub API 연동 (PM-4 · PO-12 완성)

현재: commitService · commitRepository 완성. GitHub API 폴링 로직만 미구현.
다음 작업: settings.githubToken · settings.githubRepo를 읽어 GitHub REST API 폴링 후 commitService.upsertBatch() 호출.
