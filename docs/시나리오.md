## 0. 이 문서가 뭔가

Pharos는 **시나리오 기반 설계**. 즉 "사용자가 처할 수 있는 12가지 상황"을 먼저 정의하고, 각 상황에서 시스템이 어떻게 동작하는지 UC(Use Case) 로 풀어쓴 것.

**중요한 결정**:
- **MVP는 시나리오 1~4만**. 나머지 8개는 v2.
- **시나리오 9 (애자일), 12 (유지보수) 는 아예 제외**.

이 문서를 읽으면:
- 우리가 만들고 있는 게 어떤 사용자 흐름인지 알 수 있고
- MVP 범위가 어디까지인지 명확히 보이고
- 각 UC가 무엇을 하는지 알 수 있음

노션이 outdated(12일+ 이전 기준) 이라 헷갈리니 **이 문서를 1차 진실** 로 봐.

## 1. Pharos 정체성 (먼저 짚기)

**한 줄**: 옵시디언 플러그인 형태의 AI 프로젝트 매니저.
**대상 사용자**: 초보 팀장 (캡스톤·사이드 프로젝트·소규모 팀에서 떠밀려 PM 된 사람).
**핵심 가치**: "회의록 한 장 던지면 AI가 분석해서 Task 만들고 일정 그려줌."

**플랫폼 결정**:
- ✅ Obsidian 플러그인 (확정)
- ❌ 별도 웹앱 (폐기됨)

**왜 Obsidian?**:
- 이미 .md 기반 작업 환경
- 팀이 Obsidian 사용 중
- 데이터 주권 (사용자 Vault에 그대로 저장)

자세한 차별점·기능은 [소개.md](../소개.md) 참고.

## 2. 서비스 5단계 전체 흐름 (가장 중요, 먼저 머릿속에 박기)

```
[1단계: 초기화]
├── 프로젝트 보고서 입력 (주제, 기능, 컨셉)
├── 팀원 정보 (이름·기술스택·고정 가용시간)
└── 고정 회의 시간 설정 토글
    ├── ON → 팀원 가용시간 입력 → AI 교집합 → 추천
    └── OFF → PO가 직접 입력
              ↓
[2단계: 기획 주간]
├── AI가 기획 로드맵 자동 생성
└── 매 고정 회의마다 사이클:
    1. 회의 주제 생성 (PO 버튼) — 이전 회의록·로드맵 진행도 종합 → 주제 추천
    2. 자료 수집 — Tavily로 외부 자료 검색 → 회의 자료 페이지에 저장
    3. 캘린더 네비게이션 — 날짜 클릭 → 주제·자료 열람
    4. 회의 후 회의록 작성 → AI 분석 → 다음 주제 재료
              ↓
[3단계: 개발 주간 전환]
├── 회의록 누적 → AI가 개발 로드맵 생성
├── 팀원 기술스택 매칭 → 업무 자동 할당
└── 로드맵 (큰 그림) + 간트차트 (팀원별 일정 상세)
              ↓
[4단계: 개발 주간 (매일)]
├── 개인 타임라인 — 본인 업무·체크리스트
├── 진행도 페이지 — 매일 자정 자동 갱신:
│   ├── GitHub 커밋 매칭 (feat(TASK-XXX): 패턴)
│   └── 체크리스트 결과
└── 회의 사이클은 2단계와 동일
              ↓
[5단계: 환경 시나리오 (예외 분기)]
└── 12개 시나리오 — 위 일반 흐름에서 분기되는 변형
```

**모든 시나리오는 이 5단계 어딘가에서 시작·분기**. 각 시나리오 = "1~4단계의 정상 흐름이 어떻게 변형되는가".

## 4. MVP 시나리오 4개 — 디테일

### 시나리오 1: 기획 완료 상태에서 시작

**상황**: 팀이 이미 기획서·아이디어가 어느 정도 잡힌 상태에서 Pharos 도입.

**흐름**:
1. PO가 새 프로젝트 생성 (보고서·기능·컨셉 입력)
2. 팀원 초대 → 각자 기술스택·고정 가용시간 입력 (PM-1)
3. AI 교집합 또는 PO 수동으로 고정 회의시간 확정 (PO-1-1)
4. AI가 기획 로드맵 생성 (PO-1)
5. 정해진 기획 기간 동안 회의 사이클 반복 (PO-2 → PO-3 → 회의 → PO-5)
6. 기획 종료 시점에 개발 로드맵 자동 생성 (PO-6)
7. 개발 진행 시작

**특이점**: 가장 표준 흐름. 다른 시나리오는 이걸 변형.

### 시나리오 2: 고정 회의 시간이 이미 정해진 경우

**상황**: 팀이 이미 "매주 화요일 19시 회의" 같은 식으로 정해놓은 상태.

**흐름**:
1. PO가 새 프로젝트 생성 시 **고정 회의 토글 OFF** 선택
2. 요일·시간 직접 입력 (when2meet 입력 단계 스킵)
3. 팀원은 가용시간 입력 안 함 (PM-1 의 가용시간 부분 스킵, 기술스택만 입력)
4. 이후 시나리오 1과 동일

**핵심**: 1단계 입력 흐름의 분기. 나머지는 동일.

### 시나리오 3: 기획 기간이 제한된 경우

**상황**: "2주 안에 기획 끝내자" 같은 짧은 기획 기간.

**흐름**:
1. 시나리오 1과 동일하게 시작
2. PO가 기획 마감일을 짧게 설정 (예: 시작 후 14일)
3. AI가 기획 로드맵 생성 시 기간 제약 반영 → **압축 모드**:
   - 기획 단계를 적은 수의 phase로 나눔
   - 주제·자료 수집을 빠르게 (자동화 비중 ↑)
4. 회의 사이클이 빠르게 회전 (주 2회 등)
5. 마감일 도달 → 자동으로 개발 단계 전환 안내

**핵심**: 시간 압박 상황에서 기획을 빠르게.

### 시나리오 4: 특정 주차까지 프로토타입 필수

**상황**: "12월 1일까지 동작하는 프로토타입 있어야 함" 같은 데드라인.

**흐름**:
1. 시나리오 1로 시작
2. PO가 마감일 외에 **프로토타입 데드라인** 별도 입력 (예: 11월 25일)
3. 시스템이 프로토타입 데드라인 N일 전부터 경고 알림:
   - 단순 규칙: `프로토타입 D-N + P0 미완료 항목 있음` 시 알림
4. Dashboard에 데드라인 카운트다운 + 미완료 P0 Task 강조 표시
5. 데드라인 D-day 가까울수록 알림 빈도 증가

**핵심**: AI가 자동으로 일정 조정하지 **않음**. 단순 알림만 (회의 합의 — 자동 일정 조정은 v2).

## 5. v2 시나리오 8개 — 디테일 (지금 구현 X, 나중에 만들 것들)

학기 안에 구현 안 하지만 **설계할 때 호환 가능하게**. 지금 코드는 이 시나리오들이 추가돼도 큰 변경 없이 동작해야 함 (확장 가능 구조).

### 시나리오 5: 팀원 중도 이탈

**상황**: 프로젝트 진행 중 팀원 한 명이 그만둠 (사정·갈등·전공 변경 등).

**흐름**:
1. PO가 팀원 관리 화면에서 그 사람 카드의 "이탈 처리" 버튼 클릭
2. 시스템이 확인 다이얼로그: "이 팀원이 담당하던 업무 N건이 미배정 상태가 됩니다. 진행하시겠습니까?"
3. PO 확인
4. 이탈 팀원의 상태가 "비활성" 으로 전환 (이름은 "유석(비활성)" 식)
5. 그 사람이 담당하던 모든 Task 의 assignee 가 "미배정" 으로 변경
6. Hocuspocus 토큰이 무효화돼 더 이상 동기화 받지 못함
7. PO 에게 안내 다이얼로그: "미배정 업무 N건이 있습니다. 일정 재조정(PO-13) 하시겠습니까?"
8. Dashboard 에 "⚠ 미배정 업무 N건" 경고 카드가 상주 표시 (다음 액션 유도)

**핵심 결정사항**:
- 이탈자의 회의록·커밋·작성물은 **삭제하지 않음** — 이름만 "(비활성)" 표시
- 이탈 사유 안 물어봄 (사생활 보호)
- 다른 팀원에게 알림 안 보냄 (PO 가 직접 공지)
- 재합류는 **새 팀원처럼 처음부터** (이전 계정·이력 안 이어짐)
- 마지막 PO 가 이탈하려면 다른 멤버를 PO 로 승격 강제

**관련 UC**: PO-14 (이탈 처리), 후속으로 PO-13 (일정 재조정 옵션 안내).

### 시나리오 6: 개발 중 새로운 기능 추가

**상황**: "이 기능도 있으면 좋겠다" 같은 추가 요청 (사용자·교수·팀 자체).

**흐름**:
1. PO 가 회의에서 새 기능 결정 → 회의록 작성
2. 회의록 분석 결과에 "신규 기능 X" 가 결정사항으로 추출됨
3. PO 가 Dashboard 에서 "일정 재조정(PO-13)" 버튼 클릭
4. AI 가 현재 로드맵 분석 → 3가지 옵션 제시:
   - **옵션 A: 마감 늦추기** — 전체 마감일 +N일, 기존 Task 그대로 유지
   - **옵션 B: 기존 Task 일부 v2 이연** — 우선순위 낮은 Task N개를 v2 로 옮기고 새 기능 끼워넣기
   - **옵션 C: 야근 모드** — 마감 그대로 유지, 새 기능을 빈 시간에 강제 배치
5. PO 가 옵션 선택
6. 로드맵 갱신 + 새 Task 생성·할당 + 영향받은 팀원에게 알림

**핵심**:
- AI 자동 X — 무조건 PO 의 명시적 클릭·선택 필요
- 옵션은 정확히 3개 (PO 가 비교 가능하게)
- 새 Task 의 assignee 는 기술스택 매칭으로 자동 추천, PO 수정 가능

**관련 UC**: PO-13 (일정 재조정), PO-11 (체크리스트), PO-6 갱신.

### 시나리오 7: 기술 스택 변경

**상황**: 개발 중 "React → Vue 로 바꾸자", "PostgreSQL → MongoDB 로 바꾸자" 같은 결정.

**흐름**:
1. 회의에서 결정 → 회의록 작성
2. 분석 결과의 "기술스택 후보" 에 신규 스택이 등장
3. PO 가 PO-13 트리거
4. AI 가 영향 분석:
   - 이미 완료된 Task: 안 건드림 (rework 옵션은 v2.1)
   - 진행 중 Task: "스택 전환 충격 N% — 부분 rework 필요" 표시
   - 미시작 Task: 새 스택으로 자동 갱신
5. AI 가 옵션 제시:
   - **옵션 A: 뒤쪽만 재계산** — 미시작 Task 만 새 스택, 진행 중은 그대로
   - **옵션 B: 진행 중 Task rework** — 진행 중 Task 도 새 스택으로 다시 (마감 +N일)
   - **옵션 C: 부분만 적용** — 특정 영역 (예: 프론트만) 만 새 스택
6. PO 선택 → 로드맵·Task 갱신

**핵심**:
- "이미 끝난 건 안 건드림" 기본 원칙
- 영향 범위 시각화 (어떤 Task 가 영향받는지 표시)
- 팀원 기술스택 정보 (PM-1) 활용해서 새 스택 가능자 자동 추천

**관련 UC**: PO-13, 팀원 기술스택 (PM-1).

### 시나리오 8: 외부 피드백 반영

**상황**: 사용자 테스트·교수·외부 멘토 피드백으로 기능 변경 필요.

**흐름**:
1. PO 가 피드백을 다음 회의 주제로 등록 → PO-2 버튼 트리거
2. 시스템이 다음 회의 주제 후보에 "피드백 반영안" 추가 추천
3. 회의 진행 → 결정사항 도출
4. 회의록 작성 → AI 분석 → 결정사항이 자동으로 Task 후보로 변환
5. PO 가 PO-11 (Task 세분화) → 체크리스트 생성
6. 영향 큰 변경이면 PO-13 호출해서 일정 재조정

**핵심**:
- 외부 피드백은 **회의 사이클 안에서 처리** (별도 긴급 흐름 X)
- PO-2 (회의 주제) 의 입력원으로 외부 피드백이 자연스럽게 들어옴

**관련 UC**: PO-2 (주제), PO-5 (회의록), PO-11 (체크리스트), 필요 시 PO-13.

### 시나리오 10: 요구사항 축소 (MVP 다이어트)

**상황**: 일정·자원 부족 발견 → 범위 축소 결정 (= "이거 다 못 해, 줄이자").

**흐름**:
1. PO 가 PO-13 트리거 with 옵션 "범위 축소"
2. AI 가 현재 로드맵의 Task 우선순위·의존성 분석
3. 옵션 제시:
   - **옵션 A: P3·P4 Task 를 v2 로 이연** — 핵심만 남김
   - **옵션 B: 기능별 부분 구현** — 각 기능을 50% 수준으로 (예: CRUD 만, 추가 기능 X)
   - **옵션 C: 특정 페이지·뷰 제거** — 범위 자체에서 빠짐
4. PO 가 선택 → 로드맵 갱신 + 이연된 Task 들은 "v2" 라벨 붙어 별도 영역으로
5. 이연된 Task 의 assignee 알림: "이 작업은 v2 로 옮겨졌습니다"

**핵심**:
- 이연 = 삭제 아님 (나중에 다시 활성화 가능)
- 우선순위 (P0~P4) 가 의사결정 핵심 입력
- 제외 기준은 PO 가 검토·수정 가능

**관련 UC**: PO-13, PO-6 갱신.

### 시나리오 11: 팀원 중간 합류

**상황**: 프로젝트 진행 중 새 팀원 합류 (충원·교체·전공 합류).

**흐름**:
1. PO 가 팀원 관리에서 "팀원 추가" 버튼 → PO-9 호출
2. 권한 프리셋 선택 (READ / WRITE / ADMIN)
3. 시스템이 일회용 토큰 생성 (24시간 만료)
4. PO 가 초대 링크 (`obsidian://pharos/join?token=...`) 신규 팀원에게 전달
5. 신규 팀원이 링크 클릭 → 옵시디언 자동 실행 + Pharos Modal 자동 오픈
6. Modal 단계별 입력:
   - 회원가입 (이름·이메일·비밀번호)
   - 기술스택 (자유 입력)
   - 고정 가용시간 (PM-1 호출, when2meet 그리드)
7. 시스템이 기존 고정 회의시간 vs 신규 가용시간 비교
8. 충돌 발견 시: 해당 회의에 "신규 팀원 = 불참" 자동 표시
9. 시스템이 온보딩 태스크 4개 자동 생성 (1주일 기한, 본인 할당):
   - □ 프로젝트 개요 문서 읽기
   - □ 기존 회의록 훑어보기
   - □ 개발 환경 설정
   - □ 첫 커밋 제출
10. PO 에게 안내: "기존 업무 일부를 신규 팀원에게 이관하시겠습니까?" → PO-13 옵션

**핵심 결정사항**:
- 자료 공개 범위: 전부 열람 가능 (편집은 권한별 차등)
- 충돌 처리: "불참" 표시만 (회의시간 재조정·가용시간 조정은 v2.1)
- PM-1 중단 시 토큰 만료 전 Modal 재진입 가능
- 멘토 시스템 X (자율 적응)
- 재합류해도 새 팀원 (이전 이력 안 이어짐, PO-14 와 일관)

**관련 UC**: PO-15 (합류), PO-9 (권한), PM-1 (가용시간·기술스택), PO-13 (이관 옵션).

### 시나리오 13: 긴급 버그·이슈 발생

**상황**: 데모 직전·중요 시점에 심각한 버그·장애 발견.

**흐름**:
1. 누군가가 "긴급 회의 추가" 버튼 (PO-4 임시 회의) → 가용시간 후보 → 빠른 시간 선택
2. 회의 진행 → 회의록 작성 → AI 분석
3. 결정사항이 "긴급 패치 작업 N개" 로 추출됨
4. PO 가 PO-11 로 체크리스트 생성 (각 Task 를 1~3 시간 단위 작업으로 쪼갬)
5. 우선순위 P0 (최우선) 자동 부여
6. 현재 진행 중인 Task 들은 일시 중단 안내
7. 패치 완료 후 원래 흐름 복귀

**핵심**:
- 긴급 회의 = 임시 회의 (PO-4) 의 빠른 버전 (가용시간 후보 중 가장 빠른 거)
- P0 우선순위가 핵심 신호
- 새 시나리오 분기 아니라 **기존 회의 사이클 + 긴급 플래그**

**관련 UC**: PO-4 (임시 회의), PO-5 (회의록), PO-11 (체크리스트).

### 시나리오 14: 프로젝트 방향 전환 (피봇)

**상황**: 큰 결정 (사용자 검증 실패, 시장 변화, 교수 피드백 등) 으로 프로젝트 방향 자체를 바꿔야 함.

**흐름**:
1. PO 가 회의 진행 → 피봇 결정 → 회의록 작성
2. 분석 결과: "방향 전환" 이 결정사항으로 등장
3. PO 가 Dashboard 에서 PO-13 광역 트리거 (옵션: "프로젝트 재정의")
4. 시스템이 옵션 제시:
   - **옵션 A: 부분 피봇** — 기능 일부 유지, 일부 폐기·신규 추가
   - **옵션 B: 전면 피봇** — 거의 처음부터 다시 (PO-0 → PO-1 → PO-6 사이클 재시작)
   - **옵션 C: 현재 마감까지만 진행 후 v2 에서 피봇** — 일단 끝내고 다음 버전에서
5. PO 선택 → 로드맵 대규모 갱신 + Task 재구성
6. 기존 회의록·작성물은 보존 (참고 자료로)

**핵심**:
- 피봇은 **시나리오 6, 7, 10 의 광역 버전**
- 기존 데이터 보존 (삭제 X)
- 결정 후 24~48시간 내 새 로드맵 확정 권장

**관련 UC**: PO-13 광역, PO-0 재정의, PO-1 / PO-6 재생성.

```
PM-2 (주간 가용시간) → PO-4 (임시 회의 일정 추천)
```

## 9. 의사결정 원칙 (회의 합의)

### "AI 자동 vs 사용자 트리거" 정책

| 케이스 | 정책 |
|---|---|
| 회의 주제 생성 | **PO 버튼 트리거만**. 자동 스케줄러 X |
| 일정 재조정 | **PO 버튼 트리거 (PO-13)**. AI 자동 변경 X |
| 회의록 분석 | 회의록 저장 시 자동 (사용자 의식 안 해도 됨) |
| GitHub 커밋 매칭 | 매일 정해진 시각 자동 폴링 |
| 데드라인 알림 | 단순 규칙 자동 (D-N + P0 미완료) |

### % 진척도 폐기 (PO-10 → PO-12)

- ❌ "이 Task 70% 완료" 같은 % 표현 모두 폐기
- ✅ 2축 모델:
  1. **효성도** (열심히도) — GitHub 커밋 활동 시간 기반
  2. **완료 체크** — 본인이 체크박스 토글

### 자동 일정 조정 (PO-7) 폐기

- 기존 PO-7 (AI 자동 지연 감지·일정 조정) **완전 폐기**
- 대신 PO-13 (사용자 주도) 로 v2 에서 처리
- MVP 에서는 **자동 변경 0%**, 사용자 명시 트리거만

### 팀원 변동 처리 단순화

- 알림 보내지 않음
- 작성물 보존 (이름만 "비활성")
- 이탈 사유 안 받음
- 재합류해도 새 팀원
- 충돌 시 "불참" 자동 표시만 (재조정 v2)
