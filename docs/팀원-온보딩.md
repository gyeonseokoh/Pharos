## 1. 이 프로젝트가 뭔가

**Pharos** = 옵시디언(Obsidian) 위에서 동작하는 AI 프로젝트 매니저.
한 줄로: "초보 팀장이 회의록 한 장 던져넣으면 AI가 분석해서 Task로 쪼개고, 팀원에게 자동 배정하고, 일정까지 그려주는 도구."

**Notion AI vs GitHub Projects 와 차별점**:
- 노션 AI: 글쓰기 보조 도구 (사용자가 부르면 도와줌)
- GitHub Projects: 이슈 칸반 (사용자가 다 입력)
- **Pharos**: 능동적 PM 에이전트 (회의록 던지면 자동 분석·생성)

자세한 차별화는 [소개.md](../소개.md) 참고.

**프로젝트 종류**: 학생 캡스톤. 학기 안에 MVP 완성이 목표.

## 2. 사용자 시나리오 (MVP 범위)

회의에서 합의한 **MVP는 시나리오 1~4** (UC 16개). 5번 이후는 v2.

### 시나리오 1: 빈 프로젝트 → 빠른 생성

1. 옵시디언에 Pharos 플러그인 설치
2. 좌측 리본 아이콘 클릭 → 빈 Dashboard
3. "새 프로젝트 생성" 버튼 → 모달 (주제·마감일·고정 회의 시간 입력)
4. Dashboard가 "프로젝트 있음 / 로드맵 없음" 상태로 전환
5. AI가 기획 로드맵 자동 생성 (지금은 2.5초 가짜 로딩 시뮬레이션)
6. Roadmap 탭에서 단계별 표시

### 시나리오 2: 정기 회의 진행

1. 캘린더에 정기 회의 자동 배치 (고정 회의 시간 기반)
2. 회의 전: AI가 회의 주제 후보 제안 (PO-2)
3. AI가 외부 자료 자동 수집 (PO-3, Tavily 검색)
4. 회의 후: 사용자가 회의록 작성 → AI가 분석 (키워드/결정사항/카테고리)
5. 회의록이 자동 분류돼 "회의록 관리" 탭의 4개 카테고리(기능 내용/개발 진행도/날짜별/기타)에 분류

### 시나리오 3: 임시 회의 추가

1. AI가 팀원 가용시간(when2meet 식 입력) 분석
2. 가용시간 교집합으로 임시 회의 시간 후보 3개 제안
3. 사용자 선택 → 임시 회의 캘린더에 추가
4. 시나리오 2의 흐름과 동일하게 진행

### 시나리오 4: 진척도 확인 + 데드라인 경고

1. 사용자가 GitHub 컨벤션 커밋 (`feat(TASK-008): ...`) 작성
2. Pharos가 매일 자동 폴링으로 GitHub에서 가져옴
3. 커밋 메시지의 TASK ID로 자동 매칭 → 해당 Task에 연결
4. Dashboard에 진행 상황 시각화
5. 데드라인 임박 시 경고 알림

**중요**: 자동 일정 조정·% 진척도 계산은 **MVP에 없음**. GitHub 매칭 + 본인 체크 2축만.

| 이름 | 역할 | 담당 |

### 핵심 폴더 패턴 이해

각 feature 폴더는 **항상 이 구조**:

```

├── domain/         ← 타입만 (로직 없음)
├── repositories/   ← 데이터 어디 있는지 추상화 (CRUD)
├── services/       ← 비즈니스 로직 Facade (UI·백엔드·AI 공용 진입점)
└── ui/             ← React 컴포넌트 + 옵시디언 어댑터
```

새 기능 추가 시 이 패턴 그대로 복붙.

```

    ↓
```

## 8. 구현된 것 — 백엔드 sync 인프라 (경석 작업)

**위치**: `feat/sync` 브랜치 (이 레포 안에 있음, 우리 main 브랜치엔 아직 머지 X)

### 핵심 파일

- `src/core/sync/ConnectionManager.ts` — Hocuspocus Provider 라이프사이클 관리
- `src/core/sync/DocumentSync.ts` — 문서 단위 실시간 동기화
- `src/ui/StatusBar.ts` — 옵시디언 하단에 동기화 상태 표시

### 동작 방식

1. 사용자가 .md 파일 열면 (`file-open` 이벤트) → 그 파일에 대한 `DocumentSync` 인스턴스 생성
2. `Y.Doc` 만들어서 `ConnectionManager` 통해 Hocuspocus 서버에 WebSocket 연결
3. `y-codemirror.next` 가 옵시디언 CodeMirror6 에디터에 yText 바인딩
4. 사용자가 키 입력하면 → CodeMirror dispatch → yCollab → yText 갱신 → 서버 broadcast → 다른 사용자에게 sync

### 통합 시점

우리 Repository/Service 작업 끝나면 `feat/sync` 머지 → 자동으로 모든 .md 파일이 동기화됨.

## 9. 구현된 것 — Repository·Service 레이어 (이번 세션)

**왜 만들었나**: 회의에서 경석이 지적한 두 문제 해결.

1. "기능 통합 → A 기능 연결하려면 일일이 찾아야 함 → Facade 패턴 필요"
2. "동기화하려면 데이터를 옵시디언 .md 파일로 저장해야 함"

### 인프라 (`src/shared/repo/`)

- `types.ts` — Repository, ChangeEvent, Disposable, EntityMeta 정의
- `frontmatter.ts` — .md frontmatter 파싱/직렬화 (옵시디언 parseYaml/stringifyYaml 활용)
- `eventBus.ts` — 도메인 이벤트 버스 (`project:created`, `meeting:updated`, `minutes:attached` 등)

### Project 엔티티 (완성, 사용 중)

- `domain/projectSchema.ts` — Zod 스키마 + 타입
- `repositories/projectRepository.ts` — 인터페이스
- `repositories/projectRepository.settings.ts` — 현재 구현체 (settings 기반)
- `services/projectService.ts` — Facade (`create`, `markPlanningGenerated`, `reset` 등)
- **사용처**: `NewProjectModal` 이 `projectService.create({ ... })` 호출, `DashboardItemView` 에서 모달 띄우면서 service 주입

### Meeting 엔티티 (완성, 미사용)

- `domain/meetingSchema.ts` — Zod 스키마 (Topic·Resource·Minutes·Analysis 임베드)
- `repositories/meetingRepository.ts` — 인터페이스
- `repositories/meetingRepository.settings.ts` — settings 기반 구현체
- `services/meetingsService.ts` — Facade (`attachMinutes`, `list`, `getById`, `listByCategory` 등)
- **사용처**: 아직 없음 (ItemView·Modal 마이그레이션 미완료)

### Service Facade 패턴이 가져다주는 것

```typescript
// 모든 호출자가 같은 한 줄로
await meetingsService.attachMinutes({
  meetingId: "mtg-001",
  content: "회의록 본문...",
  authorName: "유석"
});

// 내부에서 자동 처리
//   - 분석 시뮬레이터 호출 (또는 미래엔 LLM)
```

## 10. 안 된 것 — VaultRepository (.md 파일 진짜 저장)

**가장 큰 미해결**.

지금은 모든 데이터가 `data.json` (옵시디언 plugin data) 에 JSON으로 들어가있음. 사용자가 옵시디언에서 회의록 작성하면:

```
사용자 입력
   ↓
settings.attachedMinutes[meetingId] = { ... }
   ↓
data.json 에 통째 저장
```

**문제점**:
- 사용자가 옵시디언에서 회의록 .md 직접 편집 불가 (.md 파일 자체가 없음)
- 경석 sync 가 동기화할 .md 파일이 없음 → 동기화 불가
- 노션 같은 자유 편집 불가

**해야 할 것**: 8개 엔티티마다 `VaultRepository` 구현체 작성.

```typescript
class VaultMeetingRepository implements MeetingRepository {
  constructor(private vault: Vault) {}

  async save(meeting: Meeting): Promise<void> {
    const path = `Pharos/Meetings/${meeting.date}_${slugify(meeting.title)}.md`;
    const md = stringifyFrontmatter(meeting, meeting.minutes?.content ?? '');
    const file = this.vault.getAbstractFileByPath(path);
    if (file) {
      await this.vault.modify(file as TFile, md);
    } else {
      await this.vault.create(path, md);
    }
  }

  async getById(id: string): Promise<Meeting | null> {
    // Pharos/Meetings/ 폴더 스캔, frontmatter id 매칭, 파싱
  }

  // list, delete, watch 등...
}
```

각 엔티티마다 1~2시간. 8개 = 1주일 작업.

## 12. 안 된 것 — 사용자 자유 편집 (노션처럼)

**현재 상태**:
- 옵시디언 자체로는 사용자가 .md 만들 수 있음 (당연히)
- 하지만 우리 플러그인은 그걸 인식 X

## 16. 우리가 지향하는 방향

### 큰 그림

```
┌──────────────────────────────────────────┐
│  UI (React)                               │
│  ItemView, View, Modal — 화면만 그림      │
└────────────────┬─────────────────────────┘
                 ↓ Service 호출 (한 줄)
┌──────────────────────────────────────────┐
│  Service Layer (Facade)                   │
│  도메인 로직 + 트랜잭션 + 이벤트 발행     │
│  UI·백엔드·AI 공통 진입점                 │
└────────────────┬─────────────────────────┘
                 ↓ Repository 호출
┌──────────────────────────────────────────┐
│  Repository Layer                         │
│  데이터 어디 있는지 추상화                │
└──────┬───────┬───────┬─────────┬─────────┘
       │       │       │         │
       ▼       ▼       ▼         ▼
   Settings  Vault MD  Server    Yjs (sync)
   (지금)    (가야할 곳)   (경석)    (자동)
```

### 결정한 것 (회의 합의)

1. 사용자는 옵시디언에서 .md 파일을 직접 편집할 수 있어야 함 (노션 스타일)
2. 한 사용자가 여러 디바이스에서 동일 Vault 사용
3. 오프라인 작업 후 온라인 시 동기화
4. 팀원 동시 편집은 Yjs CRDT가 자동 해결
5. AI는 버튼/자동 트리거만 (자연어 명령 X)
6. GitHub은 폴링
7. % 진척도·자동 일정 조정 X (GitHub 매칭 + 본인 체크만)
8. 회의록 본문/Task 체크 = 절대 보존 / AI 분석 = 재생성 OK
9. **모든 .md 쓰기는 `vault.modify` 통일** (열린·닫힌 분기 X)
10. 닫힌 파일 sync 트리거는 경석 sync 모듈이 처리
11. 동기화 필터는 경석 코드에 통합 시 적용
12. workspaceId 자동 생성 (UUID), 초대 링크 포함
13. `.work.json` 메타파일로 자동 작업 중복 방지

### 핵심 패턴 (Facade)

**경석이 회의에서 짚은 문제**: "A 기능 연결하려면 일일이 찾아야 함 → Facade 패턴으로 코드 구조화 필요"

**우리 답**: 각 도메인마다 Service 클래스 1개. UI·백엔드·AI 모두 그 Service 메서드만 호출하면 끝. Service 내부에서 Repository·시뮬레이터·이벤트버스 등 모두 자동 처리.

```typescript
// 어디서든 한 줄
await meetingsService.attachMinutes({ meetingId, content, authorName });
```

이게 우리가 향하는 방향의 핵심.

## 20. 자주 묻는 질문

### Q: 어디서 시작해야 하나?

1. 이 문서 (팀원-온보딩.md) 처음부터 끝까지 읽기
2. 옵시디언에 설치해서 직접 만져보기 (시연 흐름)
3. `features/project/` 폴더 통째로 읽기 (Repository 패턴 이해)
4. 본인 영역(서버·AI·GitHub) 관련 섹션 다시 읽기

### Q: 새 기능 추가하려면?

1. `features/<name>/` 폴더 만들기
2. `domain/<name>Schema.ts` 에 Zod 스키마 작성
3. `repositories/<name>Repository.ts` 인터페이스 + `.settings.ts` 구현체
4. `services/<name>Service.ts` Facade
5. `main.ts` 에 의존성 주입 + `PharosPluginLike` 타입 추가
6. UI 에서 `plugin.<name>Service.method()` 호출

`features/project/` 또는 `features/meeting/` 그대로 복붙 후 이름·필드만 바꾸면 됨.

### Q: 모든 데이터가 data.json 에 있는데 옮겨야 하나?

지금은 settings 기반. 나중에 VaultRepository 만들 때 옮길 거. **지금 당장은 안 옮겨도 됨**.

### Q: 시뮬레이터 vs LLM 어떻게 분기하나?

지금은 시뮬레이터 직접 호출. LLM 붙으면 `settings.useRealAI` 같은 feature flag 추가 + Service 내부에서 분기.

### Q: 충돌이 두려운데 머지는 어떻게?

지금 브랜치는 `Pharos-UI/UX`. 새 작업은 `feature/<your-name>` 브랜치에서. PR 올려서 리뷰 받고 머지. (Git Flow 정착은 다음에)

### Q: 옵시디언이 한국어 파일 인식하나?

OK. `소개.md`, `팀원-온보딩.md` 같은 한국어 파일명 모두 정상 동작.

### Q: 이 문서가 outdated 면?

작성일 (2026-05-06) 확인 후 깃 history 로 최근 변경 봄. 큰 변경 있으면 이 문서 갱신해줘.
